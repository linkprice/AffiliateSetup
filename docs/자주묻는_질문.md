## 자주 묻는 질문
### Q. 구매/취소 확정 처리 시점 문의
* 링크프라이스의 정산은 실적 발생월의 익월 20일부터 익익월 5일까지 진행됩니다.
* 확정된 실적은 매월 6일에 전전월 실적을 조회하여 확인하실 수 있습니다. 
* 정산까지 3개월 이상 걸리는 머천트의 실적은 발생월의 익익월 정산에서 취소된 후 정산에 해당하는 달에 확정실적으로 생성됩니다.    
예) `육스(yoox)` 머천트는 구매월 기준 정산까지 3개월이 걸립니다.     
2025년 9월 26일 구매 시 `20250926` 실적이 생성되며 2025년 11월 6일에 해당 실적을 조회하면 `취소완료(310)` 실적으로 표시됩니다. 이후 2025년 12월 6일 조회시 `20251001` 일자로 생성된 `확정실적(210)`을 확인하실 수 있습니다.    
위 예에서 `20250926` 실적이 `20251001` 실적으로 생성된 이유는 `육스(yoox)` 머천트 컨디션입니다. 다만 정산까지 3개월 이상 걸리는 머천트 대부분이 확정실적을 정산이 끝난 달의 전전월에 포함되도록 생성하여, 매월 6일에 전전월 실적만 조회하면 전전월보다 오래된 실적도 확인할 수 있도록 진행되고 있습니다.

### Q.  API 호출 제한 문의
* 별도의 호출 제한은 존재하지 않으나 월별 조회 등 시스템 부하가 발생할 수 있어 내부적으로는 15분에 1회 가량 호출하는 것을 권장하고 있습니다.
* 사전 협의 없이 자주 호출하여 시스템 부하 발생 시 별도 사전 안내 없이 해당 어필리에이트 코드 기준으로 API 호출이 금지 될 수 있으니 이용에 유의 바랍니다.

### Q. 유니크 값 문의
* `trlog_id`는 실적의 고유값입니다. 하지만 정산 컨디션이 `실시간 실적 전체 취소 후 구매 확정 실적 재생성`인 머천트일 경우, 리워드 API로 전달된 실적은 정산 시 `취소완료(310)` 상태로 처리되며, 이 후 생성되는 `확정실적(210)`의 `trlog_id` 값과 다르기 때문에 `trlog_id`로 추적이 불가합니다.
* 실적 추적 시 변하지 않는 값은 리워드 API 기준으로 `merchant_id`, `order_code`, `affiliate_user_id`이며, 실적조회 API 기준으로는 `m_id`, `o_cd`, `user_id` 입니다.

### Q. 응답 데이터 문의

* `status`

  * 100: 일반 상태. 실적이 발생하면 부여되는 상태입니다.

  * 200: 정산 대기 상태. 내부적으로 정산 과정에서 관리되는 상태로 구매 확정인 상태가 아닙니다.    
  
  * 210: 정산 완료 상태. 실적의 커미션 지급이 확정된 상태이며 더 이상 상태가 변경되지 않습니다.

  * 300: 취소 대기 상태. 내부적으로 정산 과정에서 관리되는 상태로 취소 확정된 상태가 아닙니다.

  * 310: 취소 확정 상태. 실적의 취소가 확정된 상태이며 더 이상 상태가 변경되지 않습니다.   
  ※ 정산까지 3개월 이상 걸리는 머천트는 `취소완료(310)` 된 실적이 정산에 해당하는 달에 `확정실적(210)`으로 생성되기 때문에 실제 취소가 아닐 수 있습니다.

  * 실적발생일로부터 일부 머천트의 경우 매월 6일 이전 정산 기간 중에는 확정된 실적이 취소 상태로 조회될 수 있으니 연동 시 참고 바랍니다.

* `create_time_stamp`

  * 실적이 링크프라이스 데이터베이스에 생성된 일자로, 실제 실적 발생일과 무관합니다.
  * 실제 실적 발생일은 `yyyymmdd`입니다.

* `pur_rate`

  * 적용된 머천트 프로그램의 건당 커미션 금액 또는 커미션 비율이며, 별도 프로모션에 의한 커미션 상향 값은 포함되지 않습니다.
  * 실제 `sales` 대비 내부적으로 계산되어 지급되는 커미션은 `commission` 항목에서 확인 가능합니다.

### Q. 파라미터 문의

* 정산 완료, 취소 완료, 발생 실적 조회 시 조회 기준 일자는 실적 발생 기준일자입니다. 정산 완료, 취소 완료 시점에 대해서는 확인이 불가능합니다.

### Q. 머천트별 실적 전송 방식

* 머천트별 실적 전송 방식의 경우 [광고주 조회 API](https://github.com/linkprice/AffiliateSetup/blob/master/docs/광고주_조회_오픈_API_1.0.md#추가-정보-응답-샘플) 의 `추가 정보 응답 샘플` 항목의  `when_trans` 응답 데이터 확인 바랍니다.
* 머천트별 정산 방식의 경우 [광고주 조회 API](https://github.com/linkprice/AffiliateSetup/blob/master/docs/광고주_조회_오픈_API_1.0.md#추가-정보-응답-샘플) 의 `추가 정보 응답 샘플` 항목의 `trans_reposition`, `commission_payment_standard` 응답 데이터 확인 바랍니다.

### Q. 암호화된 회원번호(u_id)가 리워드 API로 정상 전달되는지 개발 단계에서 확인할 수 있나요?

* 리워드 API는 별도의 개발/스테이징 환경을 제공하지 않습니다.
* 정상적인 플로우(클릭 → 구매 → 실적 적재 → 리워드 API 전송)를 통해서만 확인 가능합니다.
* 개발 단계에서는 다음 방법 중 하나로 검증할 수 있습니다.
  - 링크프라이스 로그 확인 요청
  - Affiliate Center에 리워드 API URL 등록 후 링크프라이스 측 테스트 전송 요청
  - Postman 등 REST API 툴을 이용한 직접 호출 테스트

### Q. 리워드 API 테스트 버튼으로 전달되는 데이터는 고정값인가요?

* 네, 맞습니다.
* Affiliate Center의 리워드 API 테스트 버튼은  
  **항상 동일한 고정 샘플 데이터 1건만 전송**합니다.
* 다양한 케이스(커미션, 상태값, 상품 정보 등)를 테스트해야 하는 경우  
  Postman 등 외부 테스트 툴을 이용해야 합니다.

### Q. 무통장입금 등 특정 결제 수단에서 커미션이 0으로 전달됩니다.

* 일부 머천트는 실시간 실적 전송 시 커미션을 계산하지 않고 `0`으로 전달합니다.
* 이후 배치 정산 시 동일 실적이 **정상 커미션으로 재전송**될 수 있습니다.
* 따라서 다음과 같은 처리를 권장합니다.
  - `commission <= 0` 인 실적은 리워드 적립 대상에서 제외
  - 후속 전송되는 확정 실적만 적립 처리

### Q. 동일한 주문에 대해 리워드 API가 두 번 이상 호출될 수 있나요?

* 네, 가능합니다.
* 다음과 같은 경우 동일 실적이 여러 번 전달될 수 있습니다.
  - 실시간 실적 + 배치 실적
  - 커미션 미확정 → 확정 재전송
  - 정산 정책에 따른 실적 재생성
* 반드시 `trlog_id` 기준 중복 방어 로직을 구현하시기 바랍니다.

### Q. 머천트의 실적은 왜 취소 후 확정 실적이 다시 생성되나요?

* 일부 머천트는 정산 컨디션상  
  **실시간 발생 실적을 전부 취소 처리(310)한 후**,  
  정산이 완료되는 달에 **새로운 확정 실적(210)을 재생성**합니다.
* 이 경우:
  - 최초 실적과 확정 실적의 `trlog_id`는 서로 다를 수 있습니다.
  - 실제 취소가 아님에도 취소 상태로 조회될 수 있습니다.
* 실적 추적 시에는 다음 조합을 기준으로 매핑을 권장합니다.
  - 리워드 API: `merchant_id + order_code + affiliate_user_id`
  - 실적 조회 API: `m_id + o_cd + user_id`

### Q. 리워드 API와 실적 조회 API 중 어떤 데이터를 기준으로 적립해야 하나요?

* 리워드 적립은 **실적조회 API 수신 데이터 기준**으로 처리하시기 바랍니다.
* 리워드 API는:
  - 확정되지 않은 데이터
  - 적립 예정내역을 회원에게 보여줄 때 사용
  - 실적조회 API 데이터와 매칭 등 운영/정산 관리 목적의 보조 수단입니다.

### Q. 머천트별 실적 전송 방식은 어디에서 확인할 수 있나요?

* 머천트별 실적 전송 방식 및 정산 컨디션은  
  **광고주 조회 API의 추가 정보 응답 데이터**를 통해 확인 가능합니다.
* 참고 필드:
  - `when_trans`
  - `trans_reposition`
  - `commission_payment_standard`
